# 编程智能体译介稿设计文档

**目标：** 基于用户提供的中文转译内容，新建一篇符合现有博客格式的译介文章，补齐 `frontmatter`、原文链接和简短导读，并保持以转译为主。

**范围：**

- 新建 1 篇文章到 `posts/reading/ai/`
- 保留用户原始转译结构与核心表达
- 统一中文排版、标题层级和术语格式
- 补充必要的元信息与导读

**不包含：**

- 不扩展成长篇评论文
- 不生成新封面图
- 不修改与当前文章无关的文件

## 选型

### 方案 1：标准译介稿

将文章放到 `posts/reading/ai/`，使用现有译介文章的组织方式：

- `frontmatter`
- 原文链接
- 简短导读
- 转译正文
- 简短收束

这是本次采用的方案，因为最符合“转译为主，只补充 formatter 和简单导引”的要求。

### 方案 2：译介 + 长评论

在正文中显著加入作者观点和延伸分析。

未采用原因：会削弱转译主体，不符合本次目标。

### 方案 3：读后笔记

基于原文重写为原创中文文章。

未采用原因：偏离“转译为主”的要求。

## 内容设计

### 文章位置

`posts/reading/ai/20260312_how_coding_agents_are_reshaping_engineering_product_and_design.mdx`

### Frontmatter 设计

- `title`：保留原文主题，并明确这是译介稿
- `date`：使用当天日期 `2026-03-12`
- `description`：概括文章核心判断，突出“实现成本下降，评审与系统思维上升”
- `category`：使用 `reading`
- `tags`：围绕 `AI Coding`、`Coding Agents`、`Software Engineering`、`Product`、`Design`
- `slug`：与主题一致、便于 SEO

### 正文设计

1. 原文链接和来源说明
2. 3 至 5 句导读，说明为什么值得读
3. 保留用户转译的大纲与主要论点
4. 统一中英文空格、术语和列表样式
5. 结尾增加一小段“为什么这篇文章值得开发者关注”

## 风险与处理

- `cover` 在 schema 中不是必填字段，但页面构建要求必须存在，因此本次复用了站内已有的封面资源以确保构建通过
- 不改动用户转译的核心判断，只做语言顺滑和格式对齐
- 完成后通过构建命令验证 MDX 和 contentlayer 是否正常
